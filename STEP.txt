    ###  how to know if the future crewmate will activate this automate workflow
    Two mechanisms, captain:

        **1. The brief (`data/<id>/brief.md`)**
        Every crewmate gets scaffolded with `bin/fm-brief.sh`. That file contains the full contract:
        - Status-reporting protocol (when/how to write `state/<id>.status`)
        - Branch setup & worktree isolation assertion
        - Push/merge rules per delivery mode
        - Definition of done

        The crewmate reads this brief on launch — it's the launch prompt passed by `fm-spawn.sh`.

        **2. Project `AGENTS.md`**
        Any durable project-intrinsic knowledge (build steps, sharp edges, conventions) lives in the project's `AGENTS.md`. Crewmates read it at session start because their harness maps `CLAUDE.md → AGENTS.md` (symlink). Firstmate ensures this file exists via `bin/fm-ensure-agents-md.sh` on first ship task.

        ---

        **To verify a future crewmate will have what it needs:**

        ```bash
        # See what brief looks like before spawning
        bin/fm-brief.sh <id> <repo-name>
        cat data/<id>/brief.md

        # Check if project has AGENTS.md
        ls projects/<repo>/AGENTS.md
        ```

        If `AGENTS.md` is missing, the first ship task creates it. If the brief has `{TASK}` still in it, you fill that before spawning — that's the only manual step.

        So: brief = crewmate's job description; `AGENTS.md` = project's memory. Both are automatic.

[Auto Agent Development and Lavish Artifact Generation]
## Self-Sufficient Auto Development

Operate as a fully autonomous agent within `sre-onduty-ai-services`. After the initial user command (`opsx:propose <prompt>`), execute the entire development, testing, and fixing lifecycle independently. Only interrupt the user for final approvals.

### 0. Lavish Session Storage

All lavish-axi sessions for this project are stored in `./artifact/sessions/state.json`.
Always prefix lavish-axi commands with the env var so sessions persist in the repo:

```bash
LAVISH_AXI_STATE_DIR=$(pwd)/artifact/sessions npx -y lavish-axi <html-file>
LAVISH_AXI_STATE_DIR=$(pwd)/artifact/sessions npx -y lavish-axi poll <html-file>
```

Artifacts live in `./artifact/lavish/`. Sessions in `./artifact/sessions/state.json`.

### 1. Autonomous Execution Loop
When a task is initiated, autonomously complete this cycle without intermediate intervention:
* **Build & Run:** Compile/build the codebase and run via `make run` (check PID if there is exist previous service run).
* **Logging:** Persist all run logs to `./logs/app-<timestamp>.log`.
* **Smoke Testing:** Execute a smoke test using `TESTING_PROMPTS.txt`. Run a minimum of 20 mixed, randomized chat interactions, maintaining a strict 50:50 distribution between wiki and triage prompts for full capability coverage.
* **Self-Prompting:** Send test interactions via Slack MCP to channel `#hq-test-public` (Channel ID: `C0AEPFMUVPF`).
* **Review & Analyze:** Autonomously review and analyze the resulting bot outputs.
* **Self-Fix & Documentation:** Identify bugs or areas of improvement and directly apply code fixes. 
* **Spec Tracking:** When applying fixes or codebase improvements, **always** use the skill `/opsx:propose` to maintain well-documented specification changes. 
* **Artifact Generation:** Utilize `/lavish` or `lavish-axi` for artifact beautifier management, saving outputs directly to `./artifact/lavish/lavish-<project-name>-<spec-name>.html`.

### 2. GitLab Merge Request (MR) Standards
When raising MRs using the GitLab MCP tool:
* Provide a concise, high-quality description detailing the scope and impact of the changes.
* **Mandatory Footer:** Always append a footer indicating the MR was generated and signed by the AI agent (e.g., `_Signed by Claude_` or `_Signed by AI Agent_`).
